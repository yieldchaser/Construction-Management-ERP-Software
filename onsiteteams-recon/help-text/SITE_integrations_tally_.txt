Architecture Tally stays on your premises. Always. Your books never leave your network. The agent runs locally, polls our cloud over HTTPS for queued transactions, and posts them to Tally on localhost:9000 . No port forwarding, no VPN, no Tally on the cloud. Outbound-only agent. The on-prem agent makes outbound HTTPS calls to our cloud. It never opens an inbound port &#8212; your firewall configuration stays untouched. Per-machine API keys. Each agent install gets a long-lived bearer key. Revoke a key from the dashboard the moment a machine is decommissioned. Tally credentials never leave Tally. The agent uses Tally&#8217;s standard XML import on localhost:9000 &#8212; no Tally login is sent to our cloud. Idempotent retries. Network glitch mid-push? The agent&#8217;s outbox replays the acknowledgement on next tick. Tally&#8217;s duplicate-voucher rejection makes the redelivery safe. Auditable. Every push has a queue ID, a saved request envelope, and a Tally response &#8212; open the diagnostic file on any failed push to see exactly what was sent and what came back. Does this work with my Tally edition? Yes &#8212; Silver, Gold, Server, Multi-User, and TallyPrime on Cloud (TPoC). The agent only needs HTTP access to localhost:9000 . It works against any TallyPrime 2.x &#8211; 7.x installation. We&#8217;ve tested on fresh installs, multi-GSTIN setups, and Server editions with 50+ concurrent users. What happens if Tally is closed when a push arrives? The agent queues. As soon as Tally is open and reachable, it drains the queue. You don&#8217;t have to keep Tally running 24/7 &#8212; the cloud holds the queue patiently. Will it create duplicate vouchers if the agent retries? No. Every push carries an Onsite record id; the cloud refuses to re-enqueue an already-synced record. Tally itself rejects duplicate voucher numbers, which our agent classifies as &#8220;already posted&#8221; and acknowledges back as success. How does multi-GSTIN routing work? You map each of your company&#8217;s GSTINs to its registration name in Tally (F11 &#8594; GST &#8594; Set/Alter GST Details). Each voucher carries a &lt;GSTREGNUMBER&gt; tag so the line lands on the correct state&#8217;s GSTR-1 &#8212; no manual filing reshuffles. Can I review before pushing? Yes. The Push page shows every fetched transaction with pre-flight warnings (missing mappings, unresolvable GST rates, etc.). Push individual records, batches, or everything &#8212; your call. What about TDS, retentions and discounts? Each Onsite deduction maps to one of: Tally&#8217;s native TDS Module (with TDS section + nature of payment), a plain ledger credit, or &#8220;skip&#8221;. Retentions (purchase-side and subcon holdback) map the same way. Discounts split between purchase-side &#8220;Discount Received&#8221; (income) and sales-side &#8220;Discount Allowed&#8221; (expense) so your P&L stays clean. What runs in the cloud vs. on-prem? Cloud: the dashboard you log into, the voucher build engine, all Onsite-side fetching, and the queue. On-prem: a tiny Python agent (~12 MB bundle, includes its own Python interpreter) that polls the cloud and POSTs to Tally locally. Do my Tally credentials go to the cloud? No. The agent talks to Tally on localhost:9000 using Tally&#8217;s standard XML import gateway &#8212; no username or password is involved on Tally&#8217;s side, and nothing about your local Tally setup is shipped to our cloud.
